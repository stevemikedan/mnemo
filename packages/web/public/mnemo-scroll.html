01 — The foundation

What is a memory?

Every fact that matters gets stored as one memory — a single row in a SQLite table with a structured set of fields. Memories aren't free-form blobs. They have types, scopes, importance scores, and a lifecycle that mirrors how biological memory works: active, fading, dormant, gone.

The six memory types

project user feedback reference episodic semantic
project — Architecture decisions, technical choices, constraints, tooling
user — Preferences, working style, communication patterns
feedback — Corrections, things to avoid, course-changes
reference — File paths, API endpoints, code pointers
episodic — What happened, when, in which session
semantic — Concepts, patterns, domain knowledge

The lifecycle

● Active

Full confidence, returned in all queries, eligible for recall and reinforcement.

◐ Dormant

Confidence fell below 0.5. Still in the store, lower weight in recall. Recalling it revives it.

○ Archived

Rarely accessed, confidence below 0.2. Excluded from most queries unless explicitly requested.

✕ Expired

Confidence dropped below 0.05. Marked expired — not deleted, but invisible in normal queries.

Scope: where does a memory live?

global
Applies everywhere. User preferences, cross-project patterns, general feedback. Every AI session reads global memories regardless of which project it's in.
project:/abs/path
Project-specific context — architecture decisions, local config, tech choices for this codebase. Only visible when an agent is working inside this path or a subdirectory.

Scope visibility uses path-prefix matching, not simple string equality — an agent in /home/user/proj/src sees memories scoped to project:/home/user/proj without any configuration. Cross-project bleed is strictly prevented.

02 — Consolidation

The dream.
Sleep-staged memory processing.

When you trigger a dream, mnemo runs four sequential processing stages — each inspired by what the sleeping brain does: deduplicate what's redundant, link what's related, decay what's unused, and reconcile what conflicts. A full dream passes in under two seconds on a store of 35+ memories.

0
Encode (pre-processing)
If an embedding provider is configured, any memories without stored vectors get encoded now. This happens first so every subsequent stage can use cosine similarity alongside keyword overlap.
N
NREM — Deduplication
Scans all same-scope memory pairs. High word-overlap (>65%) → merge immediately. Low overlap (<30%) → skip. Middle ground → first the trained pre-screener ELM gets a look: a confident SKIP prediction avoids the LLM call entirely. Otherwise the configured LLM (fast ollama/llama3.2:3b) adjudicates. Merged memories fold the older content into the survivor's text and mark the duplicate archived — with both memories' before-state snapshotted to dream_audit first, so a bad verdict is recoverable.
R
REM — Cross-linking
Finds pairs of memories in the same scope that are thematically similar and don't already share an edge. When both memories have stored embeddings, the "related but not duplicate" band is semantic cosine 0.55–0.85 — catching paraphrase relations that share no vocabulary; pairs without vectors fall back to lexical overlap (0.25–0.65). Adds a relates-to edge. This is how the knowledge graph grows naturally over time without manual curation.
D
Decay — Lifecycle management
For each memory, computes a retention score based on base half-life (30 days), importance multiplier, and access frequency. If retention falls below the dormant threshold (0.5) → demoted. Below archive threshold (0.2) → archived. Below expire threshold (0.05) → expired. High-importance memories (≥0.9) are pinned — they never decay below active.
C
Reconcile — Contradiction detection
Finds same-scope, same-type memory pairs that share enough vocabulary (overlap > 0.2) or enough stored-embedding similarity (cosine ≥ 0.6 — so "deploys via GitHub Actions" vs "CI runs on Jenkins now" gets caught despite zero shared words) and don't already have a relationship edge. The pre-screener ELM clears confident non-conflicts first; the rest go to a high-quality LLM (claude-cli/haiku — separate from the NREM model) to judge: SUPERSEDES (newer overrides older, older gets demoted — before-state audited), CONTRADICTS (genuine conflict, both preserved with a contradicts edge), or NONE (compatible, no edge needed). Capped at 24 pairs per run to bound cost.
Split-provider design + fallback chain
NREM merge decisions use ollama/llama3.2:3b — fast, local, free, good enough for near-duplicate merge calls. Reconcile uses a stronger model (e.g. claude-cli/sonnet) for nuanced contradiction and supersession judgment, where a false positive creates a bad graph edge. A fallback chain makes an aggressive primary safe: if it's unreachable (logged out, rate-limited, offline), the next provider runs, then the built-in heuristic — never a silent skip. Each verdict records the model that actually ran, so training-label filtering stays honest and per-dream fallback counts surface in the dream log.
Adjudication log → pre-screener
Every LLM verdict (merge, contradiction, supersession) gets logged to adjudication_log with its feature vector and the model that rendered it. Those rows train the pre-screener ELMs during every dream — and once a model beats its baseline on held-out data, confident SKIP/NONE predictions skip the LLM call. Safety asymmetry by design: the ELM may only short-circuit negative verdicts; anything that would merge or demote a memory still gets LLM confirmation.
Spaced repetition
Recalling a memory — via search, ask, or any MCP tool that returns it — bumps its access_count and resets last_accessed. The decay formula reads both: accessed memories decay slower, frequently-recalled memories become nearly permanent. This is the same mechanism as spaced repetition flashcard systems, applied to structured facts.
03 — The graph

Memories connect.
A knowledge graph grows.

mnemo isn't a flat list — it's a directed graph. Every memory is a node; relationships between them are typed, weighted edges. The graph makes contradictions visible, shows which facts supersede others, and lets the dashboard render a live map of your knowledge.

↔ relates-to
Added automatically by the REM phase when two memories are thematically similar. The most common edge — it's how context clusters form. No semantic value (neither memory is preferred over the other), just "these are about the same thing."
→ supersedes
Directed: newer→older. Added by reconcile when a newer memory renders an older one outdated. The older memory's importance is dropped to ≤0.15 and its metadata gets a superseded_by pointer. Superseded facts are excluded from recall and chat grounding entirely — an agent asking "how do we deploy?" never gets the fact the system knows is stale (the dashboard browser still shows them).
⚡ contradicts
Added by reconcile when two memories conflict about the same subject but neither is clearly authoritative (e.g. two different team members' stated preferences that differ). Both are preserved — and when chat or ask retrieves one side, expandConflicts() pulls in the other and instructs the LLM to present both. mnemo doesn't pick a winner, and it doesn't let an answer silently pick one either.
⊂ derived-from
Set manually: "this memory was extracted from, or is based on, that one." Useful for tracing where a synthesized fact came from. Not set automatically.
⦿ co-occurred
Two things that appeared in the same context or session. The weakest edge type — no causal or logical relationship implied. Available for manual use; not added automatically by the dream process.

Edges are stored in a separate memory_edges table with from_id, to_id, edge_type, and a float weight (0–1). The graph is scoped: edges only form between memories in the same scope or between a project memory and a global memory — never across unrelated projects. Foreign keys + ON DELETE CASCADE keep the graph consistent when memories are deleted.

04 — Retrieval

Finding what matters.
Hybrid search, ranked by both.

mnemo uses two retrieval signals — keyword overlap and semantic similarity — fused together using Reciprocal Rank Fusion (RRF). Neither signal alone is sufficient: BM25 misses paraphrases; vectors miss rare exact terms. Together they find the right memories.

BM25 — keyword retrieval

Built from scratch in TypeScript: tokenizes memory content, builds an inverted index, scores with the BM25 formula (TF-IDF with length normalization). Fast, zero dependencies, accurate for exact-term queries.

// Score = Σ IDF(t) × TF(t,d) × (k₁+1) // TF(t,d) + k₁·(1-b+b·|d|/avgdl) // k₁=1.5, b=0.75 (standard values)

Vector — semantic retrieval

Each memory's embedding (stored as a SQLite BLOB, 768-dim with Ollama/nomic-embed-text) gets cosine-compared against the query embedding at recall time. Finds memories that mean the same thing even with completely different words.

// cosine(a, b) = (a·b) / (|a|·|b|) // Stored as Float32Array BLOBs → decode → dot product // Memory access = O(n), ~1ms for 100 memories

Reciprocal Rank Fusion (RRF)

The BM25 ranked list and the vector ranked list are merged using RRF: for each memory, its RRF score is the sum of 1/(rank + k) across both lists (k=60 to de-emphasize very high ranks). A memory that ranks well in both lists gets a strong combined score; one that ranks well in only one still surfaces but lower. The top results are returned after scope filtering and dedup.

// RRF score for memory m: const rrf = (1 / (bm25Rank + 60)) + (1 / (vectorRank + 60)); // k=60 standard; memories only in one list: rank = ∞ → contribution = 0

Then the learned reranker gets a vote

After fusion, a third signal blends in: an on-device ELM trained on recall_feedback — which memories were actually used when previously retrieved (cited by chat answers, or marked via the record_use tool). Its P(used) multiplies the fused score as score × (0.5 + P), so the model can nudge but never invert a strong BM25+vector consensus. Finally an importance boost applies, which is what sinks reconcile-demoted stale facts. Until the reranker beats its validation baseline, this stage is a byte-identical no-op.

05 — Conversational access

Ask your memories.
Multi-turn RAG chat.

The mnemo dashboard includes a grounded conversational interface. Every message re-retrieves the most relevant memories for that specific turn, then passes the full conversation history to the LLM alongside that memory context. The store is the session state — no separate chat memory is needed.

1
User sends a message
The message is appended to the in-memory conversation thread. The full thread (all prior turns) is kept — the LLM sees the entire history on every call.
2
Retrieval — per turn, with query rewriting
On follow-up turns, condenseQuery() first LLM-rewrites the latest message into a standalone query — "tell me more about that" becomes "github actions deployment pipeline" using the conversation to resolve the pronoun. Then searchHybrid() runs the same BM25+vector+reranker retrieval as a normal recall() over candidates with superseded facts filtered out, returning the top 6. This re-runs every turn — as the topic shifts, so does the grounding context.
3
Memory context injected as system prompt
First expandConflicts() checks the graph: if a retrieved memory has a contradicts edge, the other side is pulled into the set. Then chatWithMemories() formats everything into a numbered system prompt: "Answer based ONLY on the stored memories below — never invent facts. Cite memories inline as [1], [2]…" — plus a KNOWN CONFLICTS note instructing the LLM to present both sides of any contradiction. The LLM receives this context fresh each turn, not as part of the message history.
4
LLM synthesizes a grounded reply
The configured reconcile LLM (same provider as contradiction detection — claude-cli/haiku by default) generates a reply using the memory context and conversation history. It cites memories by index and signals when the memories don't cover the question rather than inventing an answer.
5
Response rendered — and the feedback loop closes
The assistant reply and source memory cards are appended to the thread (persisted in localStorage across refreshes). Conflicted sources get a ⚠ badge. Then the citations are parsed: sources the answer cited become recall_feedback positives, sources shown but uncited become true negatives — the exact training data the recall reranker learns from during the next dream. Every conversation makes retrieval a little smarter.
// packages/core/src/rag/answer.ts export async function chatWithMemories( messages: ChatMessage[], // full conversation history memories: Pick<Memory, 'content' | 'type'>[], // top-k + conflict partners chat: Chatter = llmChat, // injectable for testing conflicts: [number, number][] = [], // contradicting index pairs ): Promise<string | null> { const context = memoryContext(memories, conflicts); // numbered memories + "KNOWN CONFLICTS: [2] contradicts [5]…" const system = `You are a memory assistant. Answer based ONLY on the stored memories below... STORED MEMORIES:\n${context}`; return chat(messages, system); // full history + fresh context each turn }
Re-retrieval per turn
Unlike systems that retrieve once and cache, mnemo re-runs retrieval on every turn — and follow-ups get LLM-rewritten into standalone queries first, so "why though?" retrieves as well as a fully-specified question. As the conversation shifts topics, the grounding context shifts with it automatically.
The store is the session state
There's no separate "chat memory" that needs to be maintained or flushed. The conversation history lives in the browser (localStorage, surviving refreshes); the long-term memory lives in mnemo. They're cleanly separated — chat context for coherence, mnemo for grounding.
Same LLM as reconcile
Chat uses llmChat(), which routes through the reconcile provider (not the faster NREM provider). That means claude-cli/haiku by default — a quality model, not the lightweight merge model. The Chatter interface is fully injectable for testing without a live LLM.
06 — On-device ML

Intelligence at the edge.
AsterMind ELM + nomic embeddings.

mnemo uses the @astermind/astermind-elm package for on-device ML — no cloud, no model download, no GPU. Four ELM models train during every dream, each gated behind held-out validation: a type classifier, two consolidation pre-screeners, and a recall reranker. A fifth component (the TF-IDF vectorizer) stands by as the fallback embedding provider.

● Type classifier
A one-hidden-layer neural network with randomly-frozen weights (the ELM trick). Only the output layer is trained — via ridge regression, not gradient descent. When you call remember(content) without specifying a type, the ELM suggests one if it's confident enough (measured by the margin between top-class and runner-up probabilities). The calling agent is told what was inferred and can always override.
● Consolidation pre-screeners
Two ELMs (one per phase) trained on adjudication_log — every past LLM verdict with its cheap pair-features. During dreams, a confident SKIP/NONE prediction skips the LLM call entirely. Deliberately asymmetric: the ELM may only short-circuit negative verdicts; predicted merges and supersessions always get LLM confirmation, because a wrong positive destroys data while a wrong negative just waits a cycle.
● Recall reranker
Trained on recall_feedback: which retrieved memories actually got used — cited by chat answers (positives), shown-but-uncited (true negatives), or marked via record_use. At recall time its P(used) blends into the fused score as score × (0.5 + P) — a nudge, never a veto. This is retrieval that learns your usage patterns, entirely on-device.
○ Standby: TFIDFVectorizer
AsterMind's TF-IDF vectorizer is the fallback embedding provider — available when Ollama isn't running. Set embeddings.provider = "astermind" to activate it. With Ollama active, this code path is dormant: its snapshot file lives at ~/.mnemo/astermind-model.json but is never loaded.

How the ELM uses nomic embeddings

With Ollama active, the ELM trains on 768-dim nomic vectors stored in the database — not on TF-IDF features. At predict time, suggestType(content) calls embedText() via Ollama to get the feature vector, then passes it to the ELM's frozen hidden layer and trained output weights. AsterMind provides the ML architecture (random projection + ridge regression); nomic provides the semantic representation.

// dream() — training path await encodeForDream(store, memories); // encode via ollama/nomic trainTypeClassifier(store); // ELM trains on stored 768-dim blobs // remember() — predict path (async) const vecs = await embedText([content]); // 768-dim via ollama const pred = clf.predict(vecs[0]); // ELM predict (sync) if (pred.margin > 0.1) suggest(pred.label); // margin gate, not confidence

The ELM only persists to disk if its held-out accuracy beats the majority-class baseline. On small or single-category stores, "do nothing" is the correct outcome. The model is never trusted silently — it annotates, never overwrites.

Deep dive: the AsterMind ML scroll →

Two flywheels, both turning

Every LLM consolidation verdict feeds adjudication_log; every chat answer's citations feed recall_feedback. Both models retrain on that data during every dream — closed-form, milliseconds — and each persists only when it beats its majority-class baseline on a held-out split. Until then, behavior is byte-identical to having no model at all. The pre-screeners never train on their own predictions (source='llm' rows only), so mistakes can't compound.

Watch it happen: the dashboard's On-device learning panel shows each model's status (off / learning / live with accuracy), and the Evaluate retrieval button scores BM25-only vs hybrid vs reranked on your real feedback data (recall@5, MRR) — so "is the ML earning its keep?" is answered with measurements, not vibes. The same data is available at GET /api/ml-status and POST /api/eval.

07 — The MCP interface

What AI agents can do.
Seventeen tools over stdio.

mnemo runs as a Model Context Protocol server — a local process that speaks JSON-RPC over stdio. Claude, Cursor, and any other MCP-compatible tool connect to it and can read, write, and consolidate memories as part of their normal work. No API keys, no cloud, no configuration per-agent.

remember(content, type?, scope?, tags?, importance?)
Store a new memory. Type can be omitted — if the ELM classifier is active and confident, it suggests one. Dedup warning fires if a near-duplicate already exists (write-time check, never blocks). Tag suggestions come from KNN over stored tag co-occurrences.
recall(query, scope?, cwd?, limit?)
Hybrid BM25+vector search over the scoped store. Returns memories ranked by RRF score with semantic similarity. Logging a recall bumps access_count — the spaced-repetition reinforcement signal.
ask_memory(question, scope?)
Ask a plain-language question; get a synthesized answer grounded in stored memories. Uses the configured consolidation LLM. Returns the answer plus source citations so the agent can link back to the originating memories.
dream(scope?, cwd?)
Run the full consolidation pipeline: encode → NREM → REM → decay → reconcile. Scoped to a project or all memories. Returns stats on merges, links, decays, and reconciliations. Safe to call after every working session.
update · forget · delete_memory · link · unlink
Mutation tools. update patches any field on an existing memory. forget marks it expired (soft delete). delete_memory hard-deletes with all edges (permanent). link/unlink manage graph edges manually.
get_memory · list_memories · list_scopes · get_status · get_dream_log · record_use
Read tools. get_status returns counts by state/type/scope plus embedding provider info. get_dream_log returns the last N consolidation runs. record_use logs that a recalled memory was actually used — training signal for a future reranker.

The server runs on stdio transport — Claude Code connects to it via the MCP config in claude_desktop_config.json. Every tool handler is hardened against crashes: uncaught exceptions and unhandled promise rejections are caught and logged to stderr (never stdout, which is the JSON-RPC channel), so a single bad call can't kill the server. Config is re-read from disk at the start of each dream with a 2-second TTL memoization — changes to ~/.mnemo/config.json apply within seconds without restarting.

08 — Configuration

One file.
Everything configurable.

All configuration lives in ~/.mnemo/config.json. Edits apply within 2 seconds without restarting anything — the server uses TTL-memoized config reads. The Settings panel in the dashboard is a live UI over the same file.

{ "embeddings": { "provider": "ollama", // none | local | astermind | ollama | openai "model": "nomic-embed-text" // model name for ollama/openai }, "consolidation": { "provider": "ollama", // none | claude-cli | anthropic | openai | ollama "model": "llama3.2:3b", // LLM for NREM merge adjudication "reconcileProvider": "claude-cli", // separate provider for contradiction/supersession "reconcileModel": "sonnet", // stronger model where nuance matters "fallback": [{ "provider": "ollama", "model": "llama3.2:3b" }] // tried when primary fails; heuristic is the floor }, "ml": { "typeSuggest": { "enabled": true }, // ELM type suggestion on remember() "tagSuggest": { "enabled": true }, // KNN tag suggestion on remember() "prescreen": { "enabled": true }, // ELM skips LLM calls for confident non-conflicts "rerank": { "enabled": true } // learned reranker blends P(used) into recall }, "decay": { "halfLifeDays": 30, // base half-life for confidence "dormantThreshold": 0.5, // active → dormant below this "protectImportance": 0.9 // memories above this never decay below active } }
claude-cli provider
Reuses your existing Claude Code login — no API key required. Runs claude -p --model haiku as a child process. Perfect for local use where you already have Claude Code installed. 60s timeout per call.
ollama provider
Talks to a local Ollama server at http://localhost:11434/v1 via the OpenAI-compatible endpoint. Supports any chat model pulled via ollama pull. Cold-start note: first call after idle takes ~45s to load the model into memory; warm calls are under 1s.
anthropic / openai providers
Direct API calls with your key. apiKey can reference ${ANTHROPIC_API_KEY} from env. The Settings UI in the dashboard hides/masks the key after save. Any OpenAI-compatible endpoint works (Gemini, Groq, LM Studio) by setting baseUrl.

A complete memory system.

Everything described here — the graph, the consolidation pipeline, the hybrid retrieval, the on-device ML — runs from a single ~/.mnemo/memory.db file. No server to host, no database to provision, no infrastructure to maintain. Your AI tools just remember.
