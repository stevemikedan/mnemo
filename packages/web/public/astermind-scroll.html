One package. Four models. Two flywheels.

mnemo doesn't just store memories. It learns from them — right on your machine, every time you consolidate.

The @astermind/astermind-elm package gives mnemo its Extreme Learning Machine — and mnemo trains four of them on your data: a type classifier that labels new memories, two pre-screeners that clear obvious non-conflicts before they cost an LLM call, and a recall reranker that learns which memories you actually use. A TF-IDF vectorizer rounds out the package as the zero-setup embedding fallback.

All of it runs entirely in Node.js — the same process as the rest of mnemo. No model files to download, no GPU required, no network request. Training is closed-form (milliseconds), so all four models retrain during every dream cycle without you noticing.

What's in the package

Two primitives.
One powers four models. One stands by.

mnemo installs @astermind/astermind-elm as its only ML dependency. The ELM primitive below is instantiated four times — type classifier, two consolidation pre-screeners, recall reranker — each trained on a different slice of your data.

Tool 1 — Embeddings
○ Standby — fallback embedding provider
TFIDFVectorizer
Term Frequency – Inverse Document Frequency

Converts a piece of text into a fixed-length list of numbers by measuring how important each word is — both within that specific document and across your entire memory collection. Words that appear everywhere (like "the") get low weight. Words unique to a memory (like "Ollama") get high weight. The result is a vector: a precise numeric fingerprint of the content.

  • Built-in stemming: "containers" and "container" map to the same token
  • Bigrams: captures "cold start" as a phrase, not just two separate words
  • L2-normalized: all vectors have the same length, so cosine similarity is a clean 0→1
  • Vocabulary capped at 1024 tokens — deterministic across runs
  • Refitted on the whole store every dream cycle (corpus-consistent vectors)
Tool 2 — Classification
● Active — four trained instances
ELM
Extreme Learning Machine

A one-hidden-layer neural network with a twist: the hidden weights are randomly set and never adjusted. Only the output layer is trained — and that's just linear algebra (ridge regression). This makes training almost instant (milliseconds, not minutes), completely deterministic with a fixed seed, and surprisingly accurate for small datasets with clear patterns.

  • Random hidden weights (frozen) + learned output weights (ridge regression)
  • Trains in milliseconds — all four models retrain every dream cycle
  • Closed-form solution: no gradient descent, no learning rate tuning
  • Seeded for reproducibility — same data, same model, every time
  • Type classifier · NREM pre-screener · reconcile pre-screener · recall reranker
Where each model lives in mnemo

Five places.
Five jobs.
Zero overlap.

01
TFIDFVectorizer → astermind embedding provider
packages/core/src/rag/astermind.ts · runs during dream()
+
When you set embeddings.provider = "astermind" in your config, mnemo uses the TFIDFVectorizer as its vector embedding engine. During every dream cycle, it reads every memory from the store, fits a fresh vectorizer on that entire corpus, then re-embeds and stores every memory's vector as a SQLite BLOB.

At query time (when you search), the same persisted vectorizer is loaded and used to embed your search query — guaranteeing that query vectors and index vectors live in the same mathematical space.

This is the "astermind" alternative to using Ollama's nomic-embed-text. It's faster and requires zero extra setup, but captures lexical similarity only — not the semantic relationships a transformer model understands.
  • Config: embeddings.provider = "astermind"
  • Runs in: fitAndEncodeCorpus(store)
  • Called from: encodeForDream() inside dream()
  • Persists to: ~/.mnemo/astermind-model.json
  • Query path: astermindEmbed([query])
  • Fallback: cosine returns 0 on dim mismatch → gracefully falls back to BM25 only
02
ELM → memory type classifier
packages/core/src/ml/elm-classifier.ts · runs on dream() + remember()
+
The ELM classifies what kind of memory a piece of text represents: user preference, feedback, project context, reference, episodic (event), or semantic (concept). It's trained during every dream cycle using the existing memories in your store as labeled training data. The model is persisted to disk and loaded on subsequent remember() calls.

When you call remember without specifying a type, the ELM runs your content through the trained classifier. If it's confident enough (measured by the margin between the top class and the runner-up), it annotates the response with its suggestion — which the calling agent can accept or override.

The model is never trusted silently: it only persists if it beats the majority-class baseline on a held-out validation set.
  • Config: ml.typeSuggest.enabled = true
  • Trains in: trainTypeClassifier(store) during dream
  • Predicts in: suggestType(content) on remember
  • Persists to: ~/.mnemo/elm-type-classifier.json
  • Gating: margin-based (top prob − runner-up), not raw confidence
  • Safety: model only saved if it beats always-"project" baseline + 5% edge
03
ELM → consolidation pre-screeners
packages/core/src/ml/prescreen.ts · trains + gates during dream()
+
Every LLM verdict from consolidation — merge or skip, supersedes or contradicts or neither — is logged to adjudication_log with eight cheap pair-features (word overlap, embedding cosine, type match, age delta…). Two ELMs train on those rows during every dream: one for NREM dedup, one for reconcile.

Once a model beats its baseline on held-out verdicts, it starts pre-screening: a confident SKIP or NONE prediction resolves the pair without an LLM call. The asymmetry is deliberate — the ELM may only short-circuit negative verdicts. A predicted merge or supersession always goes to the LLM, because a wrong positive destroys a memory while a wrong negative just waits for the next dream. And the models never train on their own predictions (source='llm' rows only), so mistakes can't compound.
  • Config: ml.prescreen.enabled = true
  • Trains in: trainPrescreener(store, phase) during dream
  • Gates in: prescreenPair(phase, a, b) before each LLM call
  • Persists to: ~/.mnemo/elm-prescreen-{nrem,reconcile}.json
  • Safety: negative verdicts only — merges/supersessions always LLM-confirmed
  • Observability: skipped-call counts land in the dream log stats
04
ELM → recall reranker
packages/core/src/ml/reranker.ts · trains during dream(), scores every recall
+
When a chat answer cites sources [1], [3] — those memories were useful. The ones shown but never cited weren't. Both signals land in recall_feedback automatically (the record_use MCP tool adds agent-side positives too).

The reranker ELM trains on that data — features like BM25 score, embedding cosine, importance, recency, and access count — and at recall time blends its P(used) into the fused ranking as score × (0.5 + P). A nudge, never a veto: it can at most halve or 1.5× a score, so it cannot invert a strong lexical+semantic consensus. This is AsterMind's "retrieval that learns" idea applied to mnemo's chat: the LLM still writes the answers, but the ELM decides what they're grounded in — and gets better at it with every conversation.
  • Config: ml.rerank.enabled = true
  • Trains in: trainReranker(store) during dream
  • Scores in: applyReranker(results) inside recall + searchHybrid
  • Persists to: ~/.mnemo/elm-reranker.json
  • Measured by: POST /api/eval — BM25 vs hybrid vs reranked on real feedback
  • Fallback: no validated model → ranking is byte-identical to plain RRF
05
The silenced() wrapper — a quiet fix for a loud problem
packages/core/src/ml/persist.ts · wraps every AsterMind call
+
The AsterMind library logs diagnostic messages to console.log and console.error during construction, training, and prediction. Normally that's harmless. But mnemo runs as an MCP (Model Context Protocol) server — a process where stdout must be exclusively clean JSON-RPC messages. A single stray log line corrupts the protocol stream, causing the host (Claude, Cursor, etc.) to disconnect.

The solution is silenced(fn): a tiny wrapper that temporarily replaces console.log and console.error with no-ops, calls the function, then restores them — whether or not the function throws. Every AsterMind call in mnemo goes through this wrapper. It's six call sites, all wrapped, no exceptions.
  • Problem: AsterMind logs to stdout → corrupts MCP stdio JSON-RPC stream
  • Fix: swap console.log/error to no-ops → call fn → restore, even on throw
  • Wraps: TFIDFVectorizer construction, vectorize, ELM construction, trainFromData, loadModelFromJSON, predictProbaFromVector
  • Location: persist.ts:silenced() — 6 active call sites
The dream() pipeline

Every night,
mnemo learns.

When you trigger a consolidation ("dream"), AsterMind runs in two back-to-back passes: first the TFIDFVectorizer refits on all your memories and stores their embeddings; then the ELM trains a fresh type classifier on those same memories. Both happen before any deduplication or cross-linking begins.

The ELM model is only written to disk if it performs better than the naïve baseline on a held-out slice of your data. If your store is too small, or too single-category, the model stays dark — which is the correct outcome.

dream.ts 
// dream() — runs during consolidation

// 1. Encode all memories into vectors
await encodeForDream(store, memories)
  // → if provider === 'astermind':
  //   fitAndEncodeCorpus(store, maxVocab)
  //   silenced(() => new TFIDFVectorizer(docs))
  //   writes vectors to SQLite BLOBs
  //   saves ~/.mnemo/astermind-model.json

// 2. Train the ELM fleet (each gated on held-out validation)
trainTypeClassifier(store)         // labels → type suggestions
trainPrescreener(store, 'nrem')     // past LLM verdicts → skip-gates
trainPrescreener(store, 'reconcile')
await trainReranker(store)          // citation feedback → recall ranking
  // → each: 80/20 seeded split, silenced() training
  // → only saved if accuracy > baseline+5%

// 3. NREM dedup — pre-screener clears confident SKIPs first,
//    LLM adjudicates the rest; mutations audited to dream_audit
await runNREM(store, graph, memories)
await runREM(store, graph, memories)
The remember() path

Every memory you
store, classified.

When an AI agent calls remember(content) without specifying a type, mnemo runs suggestType(). This loads the persisted ELM, embeds the new content, and asks the model what category it belongs to.

If the ELM is confident enough (the margin between top and runner-up exceeds a threshold), the suggestion is used and the agent is told. If it's a near-tie — or there's no trained model yet — mnemo quietly defaults to "project". The agent can always override.

write.ts · suggestType() 
// remember(content) — called by AI agent

let finalType = type // undefined if omitted

if (!finalType && config.ml?.typeSuggest?.enabled) {
  const suggested = await suggestType(content)
  // → loadElmClassifier(persisted JSON)
  // → embedText([content]) via configured provider
  // → silenced(() => elm.predictProbaFromVector(vec))
  // → {label, confidence, margin, proba}
  // → null if margin < minMargin threshold

  if (suggested) {
    finalType = suggested.type
    note += `Type inferred as \`${suggested.type}\`
             (margin ${suggested.margin.toFixed(2)})
             — pass \`type\` to override.\``
  }
}

finalType ??= 'project' // safe default
store.create({ content, type: finalType, … })

Things we discovered the hard way

Quirks & surprises
from real use.

The stdout problem

AsterMind's library logs to console.log on every constructor, training run, and model load. In a normal Node app, this is harmless noise. In an MCP stdio server, it's fatal: a single stray line corrupts the JSON-RPC protocol and disconnects your AI host.

The fix is silenced(fn) — six lines of code that swap console methods to no-ops, call the function, and restore them. Every single AsterMind call site in mnemo is wrapped. No exceptions. Missing one breaks the MCP connection.

The confidence calibration quirk

The ELM's softmax output is systematically underconfident. Even when the model classifies correctly at 98% accuracy on the holdout set, the raw top probability rarely exceeds 0.35. If you gate on "confidence > 0.75," the ELM never fires — ever — even when it's right.

The fix: use margin (top probability minus runner-up probability) instead of raw confidence. A margin of 0.10 separates a decisive prediction from a near-tie, and it's well-calibrated for this model even though absolute confidence is not. The threshold that worked in practice: minMargin = 0.1.

Corpus consistency

The TFIDFVectorizer must be fitted on the same corpus of documents at both index time (when memory embeddings are written) and query time (when your search is vectorized). If you add new memories and re-embed without refitting the vectorizer, the new vectors live in a slightly different space.

mnemo handles this by refitting on the entire store during every dream cycle — not just the new memories. It's more work per dream, but it guarantees that all stored vectors and every future query use the same vocabulary and IDF weights. Dimension mismatch (from a refit mid-session) gracefully falls back to BM25-only retrieval.

Embedding provider comparison

Where does astermind fit?

mnemo supports four embedding providers. AsterMind's TFIDFVectorizer sits in the middle: faster and simpler than a transformer model, richer than a pure hashing function.

Provider Algorithm Dim Speed Quality Requires
none BM25 only (no vectors) — Instant Keyword match only Nothing
local FNV-1a feature hashing + bigrams + L2 256 Instant Lexical (fuzzy) Nothing built-in
astermindTF-IDF TFIDFVectorizer + stemming + bigrams + L2 256–1024 Fast (~ms) Lexical + corpus-adaptive @astermind/astermind-elm
ollamaactive Transformer (nomic-embed-text, 12-layer) 768 ~1s warm / 45s cold Semantic (best) Ollama + nomic-embed-text
openai Transformer (text-embedding-3-small) 1536 ~300ms (network) Semantic (best) OpenAI API key + internet

Currently active: ollama / nomic-embed-text (768-dim transformer). The astermind provider is available as a zero-setup alternative for environments without Ollama. The ELM type classifier now trains on whichever embeddings are stored in the database — so with ollama active, the ELM gets richer 768-dim features than TF-IDF can provide.

ML
Design principles behind the integration
01
Every ML path has a graceful fallback

If the ELM has no trained model, or if confidence is below the margin threshold, or if the embedding provider is offline, mnemo's behavior is identical to having no ML at all. Type defaults to "project." Embeddings are omitted. Retrieval falls back to BM25. Nothing breaks, nothing is blocked, nothing is silently wrong.

02
The ELM only ships when it earns it

Every training run produces a held-out validation report. The ELM classifier is only persisted to disk — and only used at write-time — when it outperforms the majority-class baseline by at least 5 percentage points on the holdout set. "Do nothing" is a first-class outcome. A model that doesn't earn its keep never touches your data.

03
Suggest-only, never silent

The ELM never silently overwrites a human or agent choice. When it suggests a type, it annotates the response with the suggestion and margin score, and explicitly tells the agent it can pass type to override. The AI stays in control. The ELM is an advisor, not an authority.

04
Two flywheels, both turning

Every LLM consolidation decision lands in adjudication_log with its feature vector, verdict, and the model that rendered it (ml.prescreen.excludeModels can drop a weak adjudicator's labels from training). Every chat answer's citations land in recall_feedback — cited sources as positives, shown-but-uncited as true negatives. The pre-screeners and the reranker retrain on this data during every dream and go live the moment they beat their baselines. Using mnemo is training mnemo: consolidations teach the pre-screeners, conversations teach the reranker, and the dashboard's On-device learning panel (or GET /api/ml-status) shows exactly where each model stands.
